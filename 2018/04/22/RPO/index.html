RPO漏洞学习笔记

2018-04-22
  • 组会内容

2018-4-21协会Web组会K0rz3n学长讲解了RPO漏洞，这里简单的记录一下我对RPO的理解。

参考博客：

K0rz3n师傅的博客

什么是RPO：
  • RPO (Relative Path Overwrite)相对路径覆盖，作为一种相对新型的攻击方式,由 Gareth Heyes在2014年首次提出，利用的是nginx服务器、配置错误的Apache服务器和浏览器之间对URL解析出现的差异，并借助文件中包含的相对路径的css或者js造成跨目录读取css或者js，甚至可以将本身不是css或者js的页面当做css或者js解析，从而触发xss等进一步的攻击手段。
触发条件：

  1. Apache 配置错误导致AllowEncodedSlashes这个选项开启（对Apache来说默认情况下 AllowEncodedSlashes 这个选项是关闭的），或者nginx服务器。

  2. 存在相对路径的js或者css的引用

触发条件1：

  • 对于Apache的配置我在本地和阿里云服务器都没有没有成功，Apache2的配置文件里并没有AllowEncodedSlashes的配置内容，搜索了大多数技术文档，给出的方法是在apache.conf中自己添加,但我实践后并没有成功（问题待解决）

  • 我在本地RPO的目录下创建了apache.php和nginx.php文件，内容分别是输出apache.php和nginx.php。

  • 分析：

  • 可以看到面对类似的url，Apache和Nginx的处理方式时不同的，对于apache，在没有开启AllowEncodedSlashes时服务器并不认识..%2f这个符号所以它将这个符号认成文件名的一部分，在它看来我们访问的时a目录下的..%2fapache.php文件，显然a路径不存在，..%2fapache.php文件也不存在，所以访问错误。但Nginx不同，它理解..%2f符号并能对其惊醒url解码，所以在它看来我们访问的是:

http://127.0.0.1/RPO/a/../nginx.php

而../这个符号对于服务器来说就是向前跳转一个目录，所以即使RPO目录下并没有a目录也没有关系，因为我们向前跳转了一个目录，所以我们访问的是RPO目录下的nginx.php，而在这个目录下文件是存在的

两者结合：

  • 1.跨目录读取js：

  • 首先我在RPO目录下创建index.php，访问后会加载本目录下的a.js，注意这个a.js前面没有/（斜杠）代表是相对路径。

  • 文件内容：







    <?php

    echo "js in test folder";

    ?>

  • 与index.php同目录下的test文件夹中有a.js，一但被调用就会弹出对话框alert(“Read file successfully”)

  • 我们访问locallhost/RPO/test/..%2findex.php

  • 发现本来只能读取和自己同一目录路下的a.js的index.php成功的读取了test目录下的a.js。

  • 分析：

  • 1.我们想服务器发送我们的请求http://localhost/RPO/test/..%2findex.php

  • 2.Nginx对..%2f进行解码，所以服务器实际认为我们访问的是http://localhost/RPO/index.php

  • 3.而浏览器根据URL路径处理index.php中引用的相对地址的脚本，可在浏览器眼中，我们访问的是test/..%2findex.php文件，所以其加载的也是test目录下的a.js，即使..%2findex.php文件根本不存在。
