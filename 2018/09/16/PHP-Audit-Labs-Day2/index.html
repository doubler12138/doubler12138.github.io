这段代码对传入的nextSlide参数进行了两次过滤，一次是在第8行使用了Twig模板的escape过滤器对它进行了过滤，实际上这里的escape过滤器是使用PHP的内置函数htmlspecialchars来实现的，这不是重点。重点是第20行，使用了filter_var函数对其进行了一次过虑。

filter_var函数：
语法：filter_var(variable,filter,option)

参数 描述
variable 必需。规定要过滤的变量
filter 可选。规定要使用的过滤器的ID
options 规定包含标志/选项的数组。检查每个过滤器可能的标志和选项

题目代码中使用的是FILTER_VALIDATE_URL过滤器，它拥有四个可能的标识：

  • FILTER_FLAG_SCHEME_REQUIRED - 要求 URL 是 RFC 兼容 URL。（比如：http://example）
  • FILTER_FLAG_HOST_REQUIRED - 要求 URL 包含主机名（http://www.example.com）
  • FILTER_FLAG_PATH_REQUIRED - 要求 URL 在主机名后存在路径（比如：eg.com/example1/）
  • FILTER_FLAG_QUERY_REQUIRED - 要求 URL 存在查询字符串（比如：eg.php?age=37）

结果：

当没有设置函数的第三个参数时，对于只要满足四种可能标志之一的变量都能通过过滤。而且filter_var的URL过滤非常的若，它只是单纯的从形式上检测并没有检测协议，所以像xxx://xxx.xxx这样的变量也是能通过过滤的，这就是这个函数最大的缺陷。

漏洞利用：

当我们了解到filter_var函数的缺陷后，我们就可以利用一些伪协议来进行绕过。对于这道题，我们可以使用javascript伪协议来绕过，从而触发xss。我们使用payload：

?nextSlide=javascript://comment%250aalert(1)

很显然这个值满足了filter_var对URL过滤的第一种可能的标志。但是，实际上javascript:是协议头，//是javascript中的单行注释，comment是被注释掉的内容，%25是被url编码过后的%，%加上0a就是换行符，这样就避免了我们后面的alert(1)被单行注释注释掉。最后我们的payload绕过了过滤，成功的被写入了echo的语句，成为了一个链接，但是当链接被点击时，最后的结果是：

题目代码在原有的filter_var的过滤以外还加了一次正则匹配的过滤，但是原理还是一样的，首先我们构造payload:xxx://";ls;"sec-redclub.com。这个payload显然是符合filter_var的过滤要求的，也符合正则匹配的要求，然后当这个值与exec函数中的字段拼接后就形成了三个命令：

  • curl “xxx://“
  • ls
  • “sec-redclub.com”

这里就涉及到命令执行了，第一个和第三个命令显然是不会执行成功的，当ls被执行后我们就能知道当前目录下的文件。当找到flagi3hEre.php后，我们用cat命令去读取。但是这里需要注意一点，cat命令和文件名之间有一个空格，如果我们直接使用空格就无法绕过filter_var函数的过滤，所以我们这里要使用<来代替空格，所以我们最后读取flag的payload如下：xxx://";cat<flagi3hEre.php;"sec-redclub.com
